Varity automatically configures deployment defaults for supported source apps and runnable Docker services.

:::note[The Varity Promise]
Usage-metered hosting still asks you to configure builds, environment variables, and separate add-on services, and the bill grows with your traffic. **Varity handles the configuration automatically** and charges a flat price for the hardware it reserves, so you can focus on building your app, not managing infrastructure.
:::

## What Makes Varity Different

Varity is built for a one-command deploy workflow from local project to live URL.

When you deploy with Varity, the platform automatically:

- **Understands your app:** figures out what your app needs without you telling it
- **Configures deployment defaults:** sets up hosting and runtime services based on your project
- **Reserves the right hardware:** reserves the hardware your app needs, so your flat monthly cost is no higher than it has to be
- **Handles hosting defaults:** Varity configures your runtime services automatically

For supported source stacks, no Docker setup is required. For other stacks, use a prebuilt runnable Docker or OCI image.

## How It Works

Varity does all the infrastructure work for you.

<Steps>

1. **You Build Your App**

   Focus on your product. Use React, Next.js, Vue, or any supported framework you like. No special Varity syntax required.

2. **Run One Command**

   ```bash
   varitykit app deploy
   ```

   That's it. No configuration files to write.

3. **Varity Figures Everything Out**

   Behind the scenes, Varity automatically:
   - Analyzes your app to understand what it needs
   - Picks the right hosting for your app (static or dynamic)
   - Configures deployment services (hosting, runtime, and detected sidecars)
   - Reserves hardware suited to your app

4. **Your App Goes Live**

   You get a live URL. Everything works automatically:
   - Hosting configured
   - Database ready when your app needs one
   - SSL certificate active
   - Static assets served from a global CDN

</Steps>

## Your App's URL

Varity gives every app a URL on `varity.app`. The shape depends on how your app is hosted:

- **Static sites** are served at a path: `https://varity.app/<your-app>/`
- **Dynamic apps** (servers, APIs, SSR, full-stack) get their own subdomain: `https://<your-app>.varity.app/`

Varity decides which one fits your project automatically. You can influence it with `--hosting` (`auto`, `static`, or `dynamic`).

## Pricing Model

Varity charges a flat monthly price for the hardware it reserves for your app. Usage-metered hosting bills by traffic, bandwidth, invocations, and build minutes, so those bills grow as your app succeeds; your Varity bill stays the same.

### The Varity Advantage

- **Reserved hardware, flat price:** your app runs on hardware reserved for it, at a price that does not change with traffic
- **No surprise overages:** no metered bandwidth, invocation, or build-minute charges
- **Included Services:** database, CDN, and SSL are included at no extra cost
- **Predictable from day one:** your bill on day 1 is your bill on day 1000

Use [Pricing and Costs](/resources/pricing) and `varity_cost_calculator` for current estimates.

## Zero Configuration Required

Traditional setups require extensive configuration:

```yaml title="Traditional: docker-compose.yml (100+ lines)"
version: '3.8'
services:
  app:
    build: .
    ports:
      - "3000:3000"
    environment:
      - DATABASE_URL=${DATABASE_URL}
      - REDIS_URL=${REDIS_URL}
      # ... more environment variables
  postgres:
    image: postgres:15
    environment:
      - POSTGRES_PASSWORD=${POSTGRES_PASSWORD}
      # ... 20+ more lines
  redis:
    image: redis:7
    # ... more configuration
  nginx:
    # ... more configuration
```

```bash title="Traditional: Required credentials"
# You have to manually set up:
export DATABASE_URL=...
export REDIS_URL=...
export STRIPE_SECRET_KEY=...
export SENDGRID_API_KEY=...
# ... more credentials
```

**With Varity:**

```bash
varitykit app deploy
```

That's it for supported source projects. You still set app-specific secrets when your code needs them.

### What Varity Auto-Configures

When you deploy, Varity automatically handles:

- Framework detection (Next.js, React, Vue, Node.js, Python, Go, static sites)
- Build configuration
- Container creation with no local Docker install required for supported source projects
- Database provisioning when your code needs it
- SSL certificates
- App URL and routing (`varity.app/<your-app>/` for static, `<your-app>.varity.app/` for dynamic)
- Global CDN for static assets
- Managed hosting defaults
- Sidecars: Postgres+pgvector, Redis, MongoDB, MySQL, Ollama, and object storage when detected in your config

**Not yet source-built:** Rust, Ruby / Rails, Elixir / Phoenix, Java / Spring, Deno, PHP / Laravel, .NET. You can still deploy these by providing a prebuilt runnable Docker image. [Request support for your framework](https://github.com/varity-labs).
