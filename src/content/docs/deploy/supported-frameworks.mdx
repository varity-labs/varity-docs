Varity automatically detects your framework when you run `varitykit app deploy`. You do not need to specify your framework manually.

## Any Public Docker Image

If your app is already packaged as a Docker image, framework detection does not apply — **any public Docker image deploys as-is**, in any language:

```bash
varitykit app deploy --image ghcr.io/acme/my-api:1.4.2 --port 8080
```

Private registries are supported from the CLI and MCP (the Developer Portal accepts public images today). See [Deploy a Docker Image](/deploy/deploy-docker-image) for all three deploy paths.

## Not Yet Supported

The following languages and frameworks are not auto-detected in the current beta. Deploying them from a repository will fail with a clear error message that includes a link to request support.

| Language / Framework | Repo auto-detection | Workaround |
|----------------------|---------------------|------------|
| Go | Not supported | [Deploy as a Docker image](/deploy/deploy-docker-image) |
| Rust | Not supported | [Deploy as a Docker image](/deploy/deploy-docker-image) |
| Ruby / Rails | Not supported | [Deploy as a Docker image](/deploy/deploy-docker-image) |
| Elixir / Phoenix | Not supported | [Deploy as a Docker image](/deploy/deploy-docker-image) |
| Java / Spring | Not supported | [Deploy as a Docker image](/deploy/deploy-docker-image) |
| Deno | Not supported | [Deploy as a Docker image](/deploy/deploy-docker-image) |
| PHP / Laravel | Not supported | [Deploy as a Docker image](/deploy/deploy-docker-image) |
| C# / .NET | Not supported | [Deploy as a Docker image](/deploy/deploy-docker-image) |

Varity also does not yet offer cron jobs, multi-region deployments, an edge runtime, or preview deploys.

If your framework is in this list, `varitykit app deploy` will return an error like:

```
ProjectDetectionError: Found go.mod - Go deployments are not yet supported.
Varity currently supports: Next.js, Astro, React, Vue, Node.js (Express/Fastify/NestJS/Koa/Hono), Python (FastAPI/Django/Flask).
Request Go support at https://github.com/varity-labs
```

No deployment is started and no charges are incurred.

<Aside type="note">
Don't see your framework? The Varity team is actively expanding support. [Open a request on GitHub](https://github.com/varity-labs) to let us know what you need.
</Aside>

## How Framework Detection Works

When you run `varitykit app deploy`, Varity scans your project directory for these signals:

| Signal | Detected As |
|--------|-------------|
| `package.json` with `next` dependency | Next.js |
| `package.json` with `astro` dependency | Astro |
| `package.json` with `react` (no Next.js) | React |
| `package.json` with `vue` | Vue |
| `package.json` with `@builder.io/qwik` or `@builder.io/qwik-city` | Qwik |
| `package.json` with `vite` | Vite SPA |
| `package.json` with `express` | Express |
| `package.json` with `fastify` | Fastify |
| `package.json` with `@nestjs/core` | Nest.js |
| `package.json` with `koa` | Koa |
| `package.json` with `hono` | Hono |
| `requirements.txt` or `pyproject.toml` | Python project |
| `package.json` with `scripts.start` and no known framework | Node.js |

Detection happens in under a second and requires no configuration from you.

## Static vs. Dynamic Hosting

Varity also decides how to host your app based on what it detects:

| App Type | Hosting Mode |
|----------|--------------|
| React or Vue SPA (no server) | Static hosting |
| Next.js with `output: 'export'` in config | Static hosting |
| Next.js with server features | Dynamic hosting |
| Express, Fastify, Koa, Hono, Nest.js | Dynamic hosting |
| FastAPI, Django, Flask | Dynamic hosting |

You can override the default with a flag:

```bash
varitykit app deploy --hosting static
varitykit app deploy --hosting dynamic
```
